Team mode

Specorator has no server. Team mode is a set of conventions layered on top of shared vault sync (Obsidian Sync or Git). This spec defines every convention, the PR sign-off detection algorithm, inline attribution format, and sync conflict handling.

1Design decision (OQ-5)

Decision: Team mode without a server = shared vault + three conventions: (1) committer handles in settings matched against Obsidian Sync authors or git commit emails; (2) PR merge as the peer gate sign-off ceremony; (3) inline <!-- @handle · date --> attribution in spec files as the audit trail. No CRDT, no WebSocket, no account required.

This decision trades some edge-case friction (Dropbox teams can't use PR detection; real-time presence is "last active" not live cursor) for zero infrastructure. That trade is correct for MVP.

2Enabling team mode

Team mode is enabled per-vault in settings (gateMode: "team"). It requires a shared vault already syncing. The plugin does not set up sync — it assumes it is already working.

SettingTypeDefaultDescription
gateModestring"solo"Set to "team" to enable team mode.
committersstring[][]List of @handles. Must match Obsidian Sync author names or git commit email prefixes.
minReviewersnumber1How many committers (excluding the feature owner) must sign off on a gate. Default 1.
Migration: When team mode is enabled on a vault with existing features, only gates crossed after enabling team mode require peer sign-off. Gates crossed in solo mode stay closed — their _meta.md records are not modified.

3Committer attribution format

Every plugin-managed write in team mode appends a committer comment to the modified line. This is the audit trail — it lives in the file, not a log.

# In a step file — OQ resolution by a peer
? Should links expire after 15 min or 60 min?
! 15 min — mobile re-send is one tap anyway.
  <!-- @arch-lead · 2026-05-01 -->

# In a step file — step hand-off
# (added to frontmatter, not body)
handed_off_by: "@arch-lead"
handed_off_at: 2026-05-01T14:32:00Z

# In _meta.md — gate crossing with peer sign-off
gates:
  - gate: define-to-build
    crossed_at: 2026-04-28T15:00:00Z
    crossed_by: "@you"
    peer_signoff: "@arch-lead"
    peer_signoff_at: 2026-04-28T14:55:00Z
    peer_signoff_method: "pr-merge"   # or "in-file"
Unknown committer: If a write carries a handle not in settings.committers, show a soft warning: "Unrecognised committer @X — add them in settings?" The action still proceeds — the warning is a nudge, not a block.

4PR sign-off flow

The Define→Build gate in team mode requires a peer review. The mechanic: a PR whose description carries the gate checklist. Merging the PR is the sign-off. No new UI ritual invented.

PR template generation

When the user clicks "Request peer sign-off" in the cockpit (gate is self-passing but peer sign-off is outstanding), the plugin generates a PR description and either opens it in the browser (if git remote is configured) or copies it to clipboard.

function generatePRTemplate(feature: Feature, gateResult: GateResult): string {
  const checks = gateResult.checks
    .filter(c => c.severity !== "off")
    .map(c => `${c.pass ? "✓" : "☐"} ${c.label}`)
    .join("\n")

  const reviewer = settings.committers
    .filter(h => h !== currentHandle())
    [0] ?? "@team"

  return `## ${feature.name} · Define → Build gate

**Reviewer:** ${reviewer}

### Gate checklist
${checks}

### Spec files
- features/${feature.slug}/01-vision.md
- features/${feature.slug}/03-rules.md
- features/${feature.slug}/04-how-it-works.md

_Merging this PR constitutes peer sign-off on the gate._
`
}

Merge-commit detection

Specorator detects a peer sign-off by watching for a commit from a listed committer that touches _meta.md in the feature folder. This is the signal that a PR was merged.

function detectPeerSignoff(feature: Feature, gate: string): boolean {
  // Check _meta.md git log for a commit by a listed committer
  // other than the feature owner, touching the feature folder

  if (!isGitVault()) {
    // Fallback: check for  comment in _meta.md
    // Added manually by peer using the in-file convention
    return hasInFileSignoff(feature, gate)
  }

  const log = gitLog(`features/${feature.slug}/_meta.md`)
  const peerCommits = log.filter(commit =>
    settings.committers.includes(commit.authorHandle) &&
    commit.authorHandle !== currentHandle()
  )

  return peerCommits.length >= settings.minReviewers
}

// In-file fallback (Obsidian Sync / Dropbox teams)
function hasInFileSignoff(feature: Feature, gate: string): boolean {
  const meta = readMeta(feature)
  const gateRecord = meta.gates.find(g => g.gate === gate)
  return !!gateRecord?.peer_signoff
  // peer_signoff is written manually by the peer editing _meta.md
}

5Peer OQ resolution

A teammate can resolve an OQ by editing the spec file directly — same mechanic as solo (? → !), plus a handle comment. No plugin action required on the peer's side. The feature owner sees the resolution on next sync.

# Peer resolves an OQ — file edit by @arch-lead
? Should links expire after 15 min or 60 min?
! 15 min — mobile re-send is one tap.
  <!-- @arch-lead · 2026-05-01 -->

# Peer escalates (can't answer) — tags the owner
? Should links expire after 15 min or 60 min? @you
<!-- @arch-lead · can't answer -->

Tag escalation detection

function isTaggedForCurrentUser(oq: OQRecord): boolean {
  // OQ line contains @{currentHandle} and is not yet resolved
  return !oq.resolved && oq.text.includes(`@${currentHandle()}`)
}

6Presence (MVP: last-active, not live)

The cockpit shows a presence strip in team mode. In MVP, presence is "last active" — read from Obsidian Sync metadata (last editor) or git log (last commit author). Not live cursor tracking.

SignalSourceRefresh
Who is in vaultObsidian Sync: app.internalPlugins.getPluginById("sync") user list (if available)On vault open. Not real-time.
What they're editingGit: last commit touching feature folder, by committer in listOn file modify event.
FallbackLast handed_off_by or found in any feature fileOn cache rebuild.
Post-MVP: Real-time presence requires either Obsidian Sync's live session API (not yet public) or a relay server. Both are out of scope for v1.

7Edge cases

Solo → team migration

Gates crossed before gateMode was set to "team" do not retroactively require peer sign-off. The plugin checks the gate record's timestamp against the settings change timestamp. If gate was crossed before team mode, it stays closed.

3+ person teams

Set minReviewers: 2 in settings. The gate check counts unique committer handles in the gate record's peer_signoff array. No UI change — the gate check label reads "2 peer sign-offs" automatically.

Committer not in list

Show soft warning. Resolution still counts. Add nudge in cockpit: "Add @handle to committers list in settings."

Vault-only teams (no git)

PR detection is unavailable. Peer sign-off is done via the in-file convention: peer edits _meta.md to add peer_signoff: "@handle" to the gate record. This is documented in the PR template as the fallback path.
