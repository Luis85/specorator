spec · 02

State machine & UX contracts

The authoritative description of every state a feature can be in, every valid transition, and the exact UI behaviour the plugin must produce. This is the most important spec page — prioritise it over all others when in doubt.

1Overview

A feature moves through 11 steps, grouped into 3 lanes. Between lanes sit 2 gates — the only places the plugin says "not yet." Every other transition is reversible within 24 hours.

DEFINE (steps 1–4)
  1 · Vision              "What are we building and why?"
  2 · Open questions      "What don't we know yet?"
  3 · Rules (EARS)        "What must always/never be true?"
  4 · How it works        "The technical approach, in prose."
  ── GATE: Define → Build ───────────────────────────────
BUILD (steps 5–8)
  5 · Check the design    "Does the design match the rules?"
  6 · Check the rules     "Are the rules still valid?"
  7 · Check the code      "Does the implementation match?"
  8 · Check security      "Any new attack surface?"
  ── GATE: Build → Ship ────────────────────────────────
SHIP (steps 9–11)
  9  · Release notes      "What changed, for humans."
  10 · Deployment log     "What ran, where, when."
  11 · Retrospective      "What would we do differently?"
Step ≠ document. Each step corresponds to one Markdown file. The file can be as short as two sentences. The plugin doesn't grade quality — it only checks for presence and structural compliance (e.g. "has at least 4 EARS rules" for step 3).

5UX contracts

These are the exact behaviours the plugin must produce in each state. "MUST" = no flexibility. "SHOULD" = default behaviour; user-configurable.

Any step file saved

contract · on file save
  • MUST re-scan the file for OQ lines (? prefix) and update the oqCounts cache.
  • MUST re-evaluate gate checks for the affected feature and update gateStatus cache.
  • MUST update last_active_at in _meta.md.
  • MUST NOT show a toast or modal unless a gate check transitions from pass → fail or vice versa.
  • SHOULD update the cockpit status bar within 300ms of save.

Step hand-off

contract · hand-off
  • MUST take a snapshot before writing any changes (DR-005).
  • MUST set status = done and handed_off_at = now on the step's frontmatter.
  • MUST advance the feature's active step to the next non-skipped step.
  • MUST show an undo banner in the cockpit for 24 hours.
  • MUST NOT advance past a gate — gate crossing is a separate action.
  • SHOULD animate the transition in 250ms.
  • MUST NOT show confetti, toast, or celebratory UI.

Gate crossing

contract · gate cross
  • MUST be unavailable if any hard-block gate check is failing.
  • MUST take a named snapshot labelled before-{gate}-gate before any write.
  • MUST update the feature's lane in _meta.md.
  • MUST log the gate crossing in gates[] in _meta.md with timestamp and actor handle.
  • MUST show an undo banner for 24 hours.
  • MUST use a 400ms transition (slower than hand-off — this is a lane change).
  • MUST NOT cross if in team mode and peer sign-off is absent.

Gate override

contract · gate override
  • MUST prompt for a one-line reason (mandatory — gate cannot be overridden silently).
  • MUST log the override in _meta.md gates[] with reason, actor, and timestamp.
  • MUST mark the gate crossing with a ⚠ in the timeline and rail.
  • MUST NOT require peer sign-off in team mode for an override — the override is a solo decision.

Undo

contract · undo
  • MUST restore all step files from the snapshot taken immediately before the action.
  • MUST restore _meta.md from the snapshot.
  • MUST be unavailable after 24 hours from the mutation timestamp.
  • MUST require a confirm step (one click, not two) before restoring.
  • MUST NOT be available for gate overrides that have already been peer-reviewed in team mode.

Cockpit empty states

contract · empty states (OQ-2 resolved)
  • MUST show the last-focused feature as default when opening the cockpit (context restoration, zero friction).
  • If all features are archived: MUST show an "all caught up" empty state with a "New feature" CTA.
  • If vault has no features at all: MUST show a "new vault" empty state with onboarding copy and a "New feature" CTA.
  • MUST NOT show a blank panel with no explanation.

Stale feature

contract · stale detection
  • MUST flag a feature as stale when now - last_active_at > settings.staleThresholdDays.
  • SHOULD show stale indicator in the feature switcher and rail.
  • MUST NOT block any action on a stale feature — stale is a nudge, not a block.

6Skip mechanics

Any step can be skipped (except steps 1 and 3, which are required). A skip is never silent — a reason is always required.

skip(step, reason):
  REQUIRE reason is non-empty string
  REQUIRE step is not 1 or 3
  snapshot(feature)           -- DR-005
  step.status = "skipped"
  step.skipped = true
  step.skip_reason = reason
  step.handed_off_at = now
  writeFile(step)
  if skippedCount(feature) >= settings.skipWarningThreshold:
    showCockpitWarning("Many steps skipped — is this intentional?")
Skip display: Skipped steps show as ◇ (diamond) in the rail, not ✓ (check). The skip reason is shown in a tooltip on hover. The step is visually distinct from done steps.
