spec · 01

Data model

All persistent state lives in Markdown files with YAML frontmatter. The plugin owns no database. This spec defines the canonical folder layout, file-naming scheme, frontmatter schema for each file type, and the in-memory cache contract.

1Vault folder layout

Default paths are user-configurable in settings. Changing paths after features exist requires a migration (not yet designed — post-MVP). The plugin warns on settings change if existing files would be orphaned.

vault-root/
├── CONSTITUTION.md          ← house rules (vault-level)
├── features/                ← configurable: settings.featuresFolder
│   └── {feature-slug}/      ← one folder per feature
│       ├── _meta.md         ← feature metadata + gate history (plugin-managed)
│       ├── 01-vision.md
│       ├── 02-questions.md
│       ├── 03-rules.md
│       ├── 04-how-it-works.md
│       ├── 05-check-design.md
│       ├── 06-check-rules.md
│       ├── 07-check-code.md
│       ├── 08-check-security.md
│       ├── 09-release-notes.md
│       ├── 10-deployment-log.md
│       ├── 11-retrospective.md
│       └── decisions/       ← optional: feature-scoped ADRs
│           └── *.md
├── archive/                 ← configurable: settings.archiveFolder
│   └── {feature-slug}/      ← identical structure to features/
├── decisions/               ← configurable: settings.decisionsFolder
│   └── *.md                 ← vault-scoped ADRs
Invariant: The plugin only scans files that match features/*/[0-9][0-9]-*.md or features/*/_meta.md. All other files in the vault are invisible to Specorator. A ? in a daily note never becomes a tracked OQ.

2Feature slug

The folder name is the canonical identifier for a feature. It is set at creation and must not change (changing it breaks all internal links and frontmatter cross-references).

slug = kebab-case(feature-name) + optional "-" + 4-digit-date
examples:
  magic-link-auth
  payment-webhooks
  user-dashboard-2026

The plugin does not enforce a date suffix but the scaffolding wizard appends one if a collision is detected.

3Step files (01-*.md – 11-*.md)

Each step file carries a mandatory frontmatter block. The plugin reads and writes status, skipped, and skip_reason. All other fields are user-editable.

FieldTypeValuesDescription
specoratorbooleantrueOpt-in marker. Plugin ignores files without this field.
stepinteger1–11Step number. Must match filename prefix.
lanestringdefine | build | shipDerived from step number. Plugin sets on scaffold; treat as read-only.
statusstringactive | done | skippedPlugin-managed. Only plugin writes this.
skippedbooleantrue | absentPresent and true when status is skipped.
skip_reasonstringfree textMandatory when skipped. One line. Shown in rail tooltip.
handed_off_atISO 8601timestampPlugin sets on hand-off. Absent if not yet handed off.
handed_off_bystring@handleSet in team mode only. Matches committer list.
 
---
specorator: true
step: 3
lane: define
status: done
handed_off_at: 2026-04-28T14:32:00Z
---
# Rules (EARS)
- WHEN user submits email THEN system sends magic link
...

4_meta.md — the feature's source of truth

Plugin-managed file. Never directly edited by the user (the plugin presents a read-only view). Contains the full feature state, gate history, snapshot index, and audit log.

FieldTypeDescription
featurestringHuman-readable feature name (from creation wizard).
slugstringCanonical identifier = folder name.
statusstringactive | archived | abandoned
lanestringdefine | build | ship | closed
created_atISO 8601Feature creation timestamp.
last_active_atISO 8601Updated on every file save within the feature folder.
gatesobject[]Array of gate crossing records (see below).
snapshotsobject[]Snapshot index — paths + timestamps. Actual snapshot files live in .specorator/snapshots/.
team_modebooleanInherited from vault settings at time of gate crossing. Recorded so gate history is accurate even if settings later change.
 
---
feature: Magic link auth
slug: magic-link-auth
status: active
lane: build
created_at: 2026-04-20T09:00:00Z
last_active_at: 2026-04-30T16:44:00Z
gates:
  - gate: define-to-build
    crossed_at: 2026-04-28T15:00:00Z
    crossed_by: "@you"
    peer_signoff: "@arch-lead"     # team mode only
    overrides: []
snapshots:
  - id: snap-001
    label: before-define-build-gate
    timestamp: 2026-04-28T14:59:00Z
    path: .specorator/snapshots/magic-link-auth/snap-001/
team_mode: false
---

5CONSTITUTION.md

A single Markdown file at vault root. Each house rule is a list item with a mandatory check method. The plugin reads this on startup and on file-save events.

# House rules

Rules that apply to every feature in this vault.

## Rules

- All UI strings must have a translation key. Check: grep for un-keyed string literals.
- Every API endpoint must have a corresponding rule in step 3. Check: count endpoints vs. EARS rules.
- No dependency added without a decision note. Check: diff package.json against decisions/.

## Retired rules
Gate check: The plugin parses every bullet in the ## Rules section. Each bullet's "Check:" clause is the human-readable description shown in the gate check list. The plugin does not execute the check automatically — it presents it to the user to confirm.

6Decision notes (ADRs)

Plain Markdown files. The plugin creates them via the "Log decision" command with a 3-field wizard. Saved to decisions/ (vault-level) or features/{slug}/decisions/ (feature-scoped). Linked from step files via wikilink.

---
decision_id: DEC-001
feature: magic-link-auth     # omit for vault-level decisions
date: 2026-04-28
status: accepted
---
# Link expiry: 15 minutes

## Context
Magic links expire. Shorter = more secure. Longer = better UX for slow email.

## Decision
15 minutes. Re-send is trivial (one tap on mobile).

## Consequences
- Re-send flow must be tested.
- Error message must explain expiry clearly.

7In-memory cache

Parsing frontmatter on every cockpit render would be too slow. The plugin maintains an in-memory cache, invalidated per-file on vault.on('modify'). The cache is rebuilt on plugin load by scanning all files matching the feature folder glob.

Cache keyValueInvalidated on
featuresMap<slug, FeatureRecord>Any file in features/*/ modified
constitutionRule[]CONSTITUTION.md modified
decisionsDecisionNote[]Any file in decisions/ modified
gateStatusMap<slug, GateResult>Any step file in the feature modified
oqCountsMap<slug, OQRecord[]>Any step file in the feature modified
Performance note: The full vault scan on plugin load should complete in <200ms for vaults up to 500 features. If a user's vault exceeds this, consider lazy-loading features not recently accessed. Benchmark on plugin load; do not block the Obsidian startup thread.

8Snapshot storage

Snapshots are full copies of the feature folder, stored in a hidden plugin-data folder. They are not visible in the Obsidian file tree.

.specorator/               ← plugin data dir (hidden from Obsidian tree)
└── snapshots/
    └── {slug}/
        └── {snap-id}/     ← full copy of features/{slug}/ at snapshot time
            ├── _meta.md
            ├── 01-vision.md
            └── ...        ← all step files at time of snapshot

Snapshot pruning: on plugin load, delete snapshots older than 24h unless they are gate-crossing snapshots (which are retained until the feature is archived). The index in _meta.md is the authoritative list of what should exist.
