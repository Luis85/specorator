Specorator
implementation spec

An Obsidian plugin that scaffolds disciplined software work: 11 steps, 3 lanes, 2 gates. Markdown-native, local-first, no server. This package covers architecture, data model, state machine, UX contracts, settings schema, commands, team mode, and key algorithms.

Target audience: external contractor with Obsidian plugin experience. Read the spec pages in order; reference the wireframes for UX detail.

Platform: Obsidian plugin (TypeScript) Storage: Markdown + YAML frontmatter Server: none Accounts: none Telemetry: none (DR-006) Min Obsidian: 1.4.0

Key decisions

DR-001 – DR-007 · accepted
ID Decision Rationale (terse) Status
DR-001 Build as Obsidian plugin, not standalone app Users already live in Obsidian. Local-first, strong plugin API, frontmatter support native. VS Code too engineering-coded; standalone too much surface area for v1. accepted
DR-002 Cockpit follows the active note (context inference) Plugin infers current feature from the path of the active Markdown file. Manual pin available as override. Empty state when no feature is in focus. accepted
DR-003 Plain-English aliases as primary labels "The rules" shown large; "requirements (EARS)" shown small. Jargon is secondary, always has a (?) popover. Power-user mode exposes orthodox vocabulary. accepted
DR-004 Markdown + frontmatter is the only persistent state No SQLite, no JSON sidecars. All state lives in YAML frontmatter on spec files. Plugin-readable, human-readable, git-friendly. Cache in plugin local storage; invalidate on mtime change. accepted
DR-005 24-hour undo on every state change (always on) Snapshot taken before every hand-off, gate cross, and file mutation. 24h window. Not configurable — this is the safety contract. Storage cost negligible (Markdown compresses well). accepted
DR-006 No telemetry, no account, no server All data stays local. Non-negotiable for v1. Obsidian users self-select for local-first. Learning happens through interviews and GitHub issues. accepted
DR-007 Sketchy lo-fi wireframes for design phase HTML wireframes with Kalam font. Iterates fast, shares cleanly, signals "not done." Will rebuild in production CSS. Not a contractor concern. accepted

Non-goals

explicit out-of-scope for v1 · say no to these
Code generation
Specorator scaffolds thinking, not code. It integrates with Cursor / Claude Code / Aider but does not compete with them.
Graph-of-everything view
Obsidian already has the graph view. Do not reinvent it.
Custom templating language
Markdown + frontmatter is enough. Template files are just notes — users can edit them directly.
Mobile app
Obsidian Mobile exists. The cockpit may not work well on mobile in v1; that is acceptable. Mobile is for reading, not for advancing gates.
AI agent inside Specorator
Core flow is deterministic — parsed frontmatter, not a model. LLM features (e.g. "summarise OQs") can layer later but must not be required.
Real-time multiplayer
Team mode is shared vault + conventions (see spec 05). No CRDT, no WebSocket, no server sync layer in v1.
