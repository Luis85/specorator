01Fundamentals

userWho Specorator is for

One person, working alone or with a small team, building software with AI assistance. They are not necessarily a software engineer — they may be a designer, a PM, a domain expert, a founder. They have ideas, they have judgement, but they have not internalized the rituals of disciplined software work: requirements, decision records, traceability, gate reviews. They want to ship something good and they suspect those rituals would help them, but the orthodox practice is daunting.

Implication: every concept must be teachable in one sentence. Jargon is not forbidden, but it must always have a one-click explainer attached. We never assume the reader has done a CS degree.

whereWhy Obsidian

The artifacts of disciplined software work are plain-text Markdown files in folders. Vision, requirements, decision notes, traceability matrices — all text. Obsidian is a beloved local-first Markdown editor with a graph view, backlinks, frontmatter, and a plugin API. It is already where many of our users keep their notes. Building Specorator inside Obsidian means we never own the user's data; we just augment what they already have.

problemThe problem we're solving

AI accelerates code generation but doesn't accelerate good judgement. Without structure, AI-assisted projects produce a lot of code very quickly that doesn't add up to a working product. The cure is not "more discipline" applied as guilt — it is a scaffold that makes the next right action obvious, and that catches mistakes before they compound.

Specorator's job is to be that scaffold. It does not write the code. It tells you what to think about next, what to write down, and what to check before moving on.

modelThe mental model

Every feature you build moves through 11 steps, grouped into three lanes: Define (figure out what), Build (figure out how, and do it), Ship (verify and release). Between lanes are gates — small checklists that prevent moving forward with unresolved questions. The 11 steps and the gates are the spine of the product.

The 11 are not arbitrary; they map to the underlying P3.express + spec-driven-development methodology the project is based on. But we never make the user say "P3" or "EARS" or "ADR" out loud unless they want to.

02Core principles

These are the constraints we hold each other to. When in doubt, re-read these.

1. Plain English first, jargon on demand.
not: "Author the EARS-formatted requirements ADR"  ·  but: "Write down the rules this feature must follow."
The orthodox terms (requirements, ADR, constitution, spec, EARS, traceability) are powerful but exclusionary. We use plain-English aliases as the primary label, with the technical term shown secondarily so users can grow into the vocabulary if they want to. A (?) next to any jargon opens a same-shape popover: one-line definition · one paragraph why · one link to read more.
2. Show one next action, not ten options.
not: a dashboard of metrics  ·  but: a coach saying "You're at step 3. Write the rules. Here's a template."
The cockpit always answers "what should I do right now?" with a single recommended action. Other actions are available, but they don't compete for attention. This is the opposite of a feature-dense IDE; it's closer to a turn-by-turn navigator.
3. The Markdown is the source of truth.
not: a database the plugin owns  ·  but: files in your vault, which the plugin reads and offers to update.
Specorator never holds state the user can't see. Frontmatter on each note encodes stage, owner, links, and gate status. If the plugin disappeared tomorrow, every artifact would still be a readable, editable, version-controlled Markdown file. This is non-negotiable.
4. Mistakes are fine. Hand-offs are reversible.
not: "are you sure?" modals  ·  but: a 24-hour undo on every state change.
Every hand-off (advance to next step, complete a gate, archive a feature) creates a snapshot. The user can undo within 24h with one click. This lets us be confident with our recommendations without making people anxious about following them.
5. Quiet by default; loud when blocking.
not: celebratory toasts  ·  but: calm progress, with attention reserved for things that actually need a decision.
No confetti, no "Great job!" No badges. The only time the UI raises its voice is when a gate is failing, a question has gone stale for >7 days, or a hand-off would silently lose information. Loudness is a budget; we spend it carefully.
6. Built for the lone builder; usable by a team.
not: a multiplayer SaaS  ·  but: a personal tool whose artifacts happen to also work in a Git-based team.
The primary user is one person. But the artifacts (Markdown + frontmatter) are git-friendly, so a team using GitHub PR review naturally inherits the discipline. Specorator never adds a server, an account, or a sync layer. Your team coordination is your repo.
7. The plugin teaches you the discipline; it doesn't replace it.
not: automate-it-all  ·  but: training wheels you can take off.
After enough use, the user should be able to explain what an EARS requirement is, when to write a decision note, and what makes a good gate. Power-user mode strips the explainers and exposes the orthodox vocabulary. We are happy when users outgrow the coach.

03Shape of the product

The cockpit is the home

Cockpit A — three panes, context-aware — is the v1 home. Left pane shows your features. Center pane answers "where am I and what do I do next" for the active feature. Right pane shows the metadata of whatever Markdown file is open. Other layouts (rail, board, feed) are modes reachable from the cockpit, not separate apps.

Decision DR-002. The cockpit is opinionated about which feature you're looking at. If you click a Markdown file in another folder, the cockpit follows. If you open a feature folder, the cockpit centers on it. The cockpit never goes blank just because you opened a stray note.

Eleven steps, three lanes, two gates

Define (1–4) → gate → Build (5–8) → gate → Ship (9–11). The two gates are the only places the system actually says "no, you can't move on." Everywhere else it suggests, recommends, and hand-holds.

Help is a popover, not a manual

Every domain term in the UI gets a (?) trigger that opens a popover with a fixed shape. We don't write a separate user manual. The popovers are the manual, surfaced exactly when needed.

04Key decisions

Numbered, dated, with the alternatives we considered. New decisions append; existing ones get superseded, never edited.

DR-001 Build inside Obsidian, not as a standalone app accepted
context
The user already has a Markdown editor they like. Building a new shell would duplicate effort and force migration.
options
(a) Standalone Electron app · (b) VS Code extension · (c) Obsidian plugin · (d) CLI only
chose
(c). Obsidian is local-first, has a strong plugin API, supports frontmatter natively, and our user persona already lives there. VS Code is too engineering-coded; standalone is too much surface area for v1.
cost
We inherit Obsidian's UI conventions, including some we'd rather replace (the ribbon, the file-tree style). We'll lean in rather than fight them.
DR-002 Cockpit follows the active note accepted
context
The plugin needs a way to know "which feature is the user thinking about right now."
options
(a) User pins a feature explicitly · (b) Cockpit infers from active note path · (c) Always show all features at once
chose
(b). Inference matches Obsidian's existing behavior and keeps the cockpit feeling like a contextual companion rather than a separate app. (a) is available as a manual override.
cost
Need a defined empty state for "no feature is in focus" — see state-empty.html.
DR-003 Plain-English aliases as primary labels accepted
context
Our users include people who are not engineers. The orthodox vocabulary (EARS, ADR, constitution, traceability) is powerful but alienating.
options
(a) Use orthodox terms everywhere · (b) Plain-English primary, technical term secondary · (c) Hide the technical terms entirely
chose
(b). "The rules" is shown big; "requirements" is shown small underneath. Users who outgrow the alias can flip to power-user mode.
cost
Twice the labels to maintain. We accept this; it pays for itself in onboarding.
DR-004 Markdown + frontmatter is the only state accepted
context
The plugin needs to know the stage of every feature, the gate status, owners, links between artifacts, etc.
options
(a) Plugin maintains a SQLite db · (b) JSON sidecar files · (c) YAML frontmatter on every note
chose
(c). Frontmatter is human-readable, git-friendly, and survives the plugin being uninstalled. Obsidian renders it cleanly.
cost
Performance: parsing many files on vault load. Mitigated by caching parse results in plugin local storage, invalidated on file mtime change.
DR-005 24-hour undo on every state change accepted
context
Hand-offs and gate completions feel scary if they're irreversible, especially while users are still learning the model.
options
(a) Confirmation modals before every change · (b) Reversible by default with 24h undo · (c) Trust users, no safety net
chose
(b). Modals slow people down without preventing real mistakes. Undo lets us be confident in our recommendations.
cost
We persist a "snapshot" per state change for 24h. Storage cost is negligible (Markdown files compress well). Cleanup on plugin load.
DR-006 No telemetry, no account, no server accepted
context
We could learn a lot from usage data. We could also build sync. Both add accounts.
options
(a) Anonymous opt-in telemetry · (b) Opt-in account for sync · (c) Strictly local, nothing leaves the user's machine
chose
(c) for v1. Obsidian users self-select for local-first; introducing an account would betray the product's premise. We'll learn through interviews and GitHub issues instead.
cost
Slower learning loop. We accept this for trust.
DR-007 Sketchy low-fi wireframes for v0.1, not Figma accepted
context
We needed to explore layout and information architecture without inviting "but the colors are wrong" feedback.
options
(a) Figma hi-fi · (b) Whiteboard photos · (c) Sketchy HTML wireframes (Kalam font, hand-drawn vibes)
chose
(c). HTML lets us iterate fast, embed real Obsidian chrome, and share a single canvas link. Sketchy aesthetic signals "this is not done yet" without making us actually undraw.
cost
Will rebuild in production-quality CSS later. The wireframes are throwaway by design.

05Trade-offs we accept

We optimize for
  • The lone, semi-technical builder
  • Long projects where memory matters
  • Local files, git, plain Markdown
  • Turn-by-turn clarity
  • Recoverable mistakes
  • Teaching, with a graceful exit
We don't optimize for
  • Multi-user real-time collaboration
  • Tiny one-shot scripts where ceremony costs more than it saves
  • Cloud-native or web-only workflows
  • Pure power-users who want a CLI
  • Project-management overhead beyond a single product line
Note. "Don't optimize for" doesn't mean "actively block." A team can adopt Specorator if they're git-disciplined; a CLI lover can ignore the cockpit. We just won't compromise the lone-builder experience to court those audiences.

06Open questions

Things we haven't decided. Each will become a numbered decision when we do.

  • OQ-1. Should the cockpit have a "command bar" inside it, or rely on Obsidian's command palette? Argues for: discoverability. Against: redundancy with native palette.
  • OQ-2. When the user opens a Markdown file not in a feature folder, should the cockpit (a) go to a "what needs you" feed, (b) show the last feature, or (c) clear with a "no feature in focus" empty state? Currently leaning (c).
  • OQ-3. Do gates require explicit sign-off, or do they auto-pass when their checklist is fully ticked? Argues for explicit: ceremony reinforces gravity. Against: friction.
  • OQ-4. How do we handle features that legitimately don't need every step? (e.g. a tiny config change.) Skip steps? Mark N/A? Have a "lite" mode?
  • OQ-5. What does "team mode" actually mean if we never add a server? Probably: better PR-review templates and per-feature CODEOWNERS hints. To be designed.

07Non-goals

Things people will ask for that we will say no to, with reason.

  • Code generation. Specorator scaffolds thinking, not files of code. We integrate with whatever AI coding tool the user prefers (Cursor, Claude Code, Aider) but don't compete with them.
  • A graph-of-everything view. Obsidian already has the graph view. We don't need to reinvent it.
  • Custom templating language. Markdown + frontmatter is enough. If users want power, they can edit the template files directly — they're just notes.
  • Mobile app. Obsidian Mobile exists. The cockpit may not work well there in v1; that's acceptable. Mobile is for reading, not for advancing gates.
  • An AI agent inside Specorator. The plugin recommends next actions deterministically, based on parsed frontmatter. We can layer LLM features later (e.g. "summarize the open questions") but the core flow doesn't depend on a model being available.
