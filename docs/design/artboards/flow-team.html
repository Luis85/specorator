Team mode — shared vault, no central server

Specorator has no backend. Team mode is Obsidian Sync (or Git) plus conventions. This flow shows what team mode actually looks and feels like, end-to-end: setup, daily use, gate sign-off, peer OQ resolution, and sync conflicts. Four acts.

Team mode complications

A
solo → team migration
Enabling team on a live solo vault
When team mode is enabled on a vault with existing features, Specorator doesn't retroactively require peer sign-off on closed gates. Only gates crossed after team mode is enabled need the peer tick. Existing _meta.md files are unchanged.
gates closed before team-mode: unaffected
gates closed after: need peer sign-off
Non-destructive migration. Old gates stay closed. New gates get the team requirement.

A handle not in the committers list
If @newperson resolves an OQ but isn't in the committers list, Specorator shows a soft warning: "unrecognised committer — add them in settings, or the resolution counts anyway." It doesn't block — it nudges.

How Specorator does team mode

choice
Shared vault, not a server

Team mode is a property of the vault, not an account. Obsidian Sync or Git is the transport. Specorator reads committer handles from the same markdown files everyone is already editing. No server, no API key, no SaaS dependency.

choice
PR as the sign-off ritual

Engineers already do PRs. Specorator generates the checklist-carrying PR description and detects the merge commit. The ceremony is real — a human reviewed and approved. The ceremony tool is one the team already trusts. No new ritual invented.

choice
Attribution in-file

Every resolution carries . This is readable in any editor, survives vault moves, doesn't require Specorator to read the history. The file is the log. (DR-004 · markdown-native)

choice
Conflicts: show, don't merge

Specorator detects conflicts and surfaces them. It never auto-merges spec text — the content of a spec is a human decision, not a diff problem. "Open in editor" is the deepest intervention the plugin makes.

Open: what's still deferred
not in MVP
Real-time presence (Obsidian Sync only shows who last edited, not who is live right now). Presence strip in S3 is "last active," not live cursor.

PR auto-detection requires a Git remote. Vault-only teams (Obsidian Sync, Dropbox) rely on the manual PR flow or treat the committer-in-file convention as the sign-off.
to revisit post-MVP
Notification when a peer resolves your OQ (currently: just the counter drop on sync). A pull notification — "arch-lead resolved 2 OQs while you were away" — might be worth a simple badge.

Multi-feature portfolio visibility: can @arch-lead see all features they're a committer on, across all vaults? Not designed. Cross-vault team mode is out of scope for now.

Team mode without a server: shared vault + conventions

Decision: Team mode is implemented entirely through shared-vault conventions, not a server. The three mechanisms are: (1) committer handles in a settings field, matched against Obsidian Sync authors or git commit emails; (2) the PR template as the peer gate sign-off ceremony, with merge-commit detection as the trigger; (3) inline <!-- @handle --> attribution in spec files as the audit trail.

Why no server: A server introduces accounts, uptime SLAs, data residency concerns, and a monetisation question on day one. All three team features (presence, sign-off, attribution) can be achieved with conventions on top of sync infrastructure teams already have. The cost is some edge-case friction (Dropbox teams can't use PR detection; real-time presence is faked). That trade is correct for an MVP.

Why the PR: The gate sign-off needs ceremony and a real human witness. The PR review already provides both. Inventing a new "review request" UI inside the plugin would duplicate the PR workflow at lower fidelity. Piggyback on the existing ritual.
