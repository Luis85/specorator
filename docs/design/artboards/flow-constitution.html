Constitution & house rules — the rules the gates enforce

The constitution (step 2) is where a project defines its own standards: coding conventions, accessibility requirements, naming rules, anything the team cares about. It's the most powerful artifact in the system — because gate checks enforce it — and the least understood. This flow covers writing it, editing it, and what happens when a feature violates it.

Editing — changing a rule mid-flight

Rules evolve. A rule added during feature 1 may need updating after feature 4. Changing a house rule mid-flight has cascading consequences — features that already passed the gate with the old rule may not pass with the new one. The plugin surfaces this explicitly.

Edge cases & design rationale

A
no constitution
Feature with no constitution
CONSTITUTION.md is optional. If it doesn't exist, the gate's "house rules respected" check is marked as exempt (not passing, not failing) — shown as a hollow circle. The feature is not penalised for not having house rules; it just doesn't get the benefit of them.
○ house rules · no constitution · exempt
Optional, not mandatory. No CONSTITUTION.md = exempt, not failing.
B
auto-verifiable rules
Rules the plugin can check itself
Some house rules can be auto-verified: "spec file must contain a 'Strings' section" is a file-content check. "Rate limiting must be documented" can be a regex. Auto-verifiable rules show a ✓ auto tag in the gate checklist; self-check rules show ✓ self.
✓ auto Strings section present
✓ self Translation keys confirmed
Auto vs self. Both count. Plugin does what it can. User covers the rest.
C
retro → new rule
Retrospective feeds constitution
When a retrospective notes a recurring problem, the coach offers to add a house rule. The new rule is drafted from the retro text and opened in CONSTITUTION.md for review before saving. The gate immediately starts checking it on future features.
retro → constitution → gate
the full learning loop
Learning loop closes here. Bug → retro → rule → gate. Never the same bug twice.
choice
One file, vault root

CONSTITUTION.md is shared across the whole vault, not per-feature. House rules are project-level conventions — they apply to everything. If a rule only applies to one feature, it belongs in that feature's step 3 (EARS rules), not the constitution. (DR-004)

choice
Start with one rule, not ten

A constitution with 20 rules from day one is a constitution nobody reads. The coach pushes for one rule to start. More rules come naturally from retrospectives and real problems. The constitution should grow from experience, not aspiration. (Principle 2)

choice
Violations are yellow, not red

A new rule added mid-flight flags existing features as needing a check — but doesn't block them. The work is already done; the team should acknowledge the gap, not be forced to undo progress. Override-with-reason is always the escape. (Principle 5)

choice
Gate history is immutable

Removing a rule from the constitution doesn't rewrite past gate audits. A feature that passed a gate with rule X should remain recorded as having passed with rule X, even if rule X no longer exists. The audit trail's value depends on its immutability.
