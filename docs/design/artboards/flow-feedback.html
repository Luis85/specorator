Error cases, user feedback & notifications

Three distinct communication concerns. Errors are things that went wrong. Feedback confirms things went right (or shows progress). Notifications are async signals that something changed while you weren't looking. Each gets a different surface and a different tone. Existing coverage (gate failures, sync conflicts, undo) is cross-referenced rather than re-illustrated.

Error cases — catalogue

Four categories of error the plugin must handle gracefully. Each has a surface (status bar dot + cockpit banner), a message (plain English, not stack traces), and a recovery action (one click to fix or acknowledge). Gate failures and sync conflicts are cross-referenced — see those flows for detail.

E1
file system
File system error
tier 2 · banner
Triggered when the plugin can't read or write a spec file — permission denied, file locked by another process, or disk full.
⚠ Can't write: 04-how-it-works.md
File is locked or permission denied. Your changes are held in memory.
changes held in memory until resolved
Changes held. Never lost. Retry or export. Snapshot taken before the failure.

Frontmatter parse error
tier 2 · banner
Triggered when YAML frontmatter in a spec file is malformed — usually caused by manual editing. The plugin can't read the feature's state.

status: in-progress
step 4 ← missing colon
cockpit paused — resumes on fix + save
Exact line shown. One-click to fix. Cockpit pauses but doesn't corrupt. Plain YAML, fixable in any editor.
E3
invalid transition
Invalid step transition
tier 2 · banner
Triggered when frontmatter is edited directly to jump steps out of order (e.g. current_step: 9 while on step 4). Plugin detects the gap and flags it.
⚑ Step jump detected: 4 → 9
Steps 5–8 would be skipped without reasons. Add skip reasons or revert to step 4.
Amber, not red. Solvable: add skip reasons. Or revert. Not a crash.

tier 2 · banner

User feedback — confirmations, progress, empty states

Feedback tells the user their action worked (or is working). Major confirmations get a cockpit banner; minor ones get a status bar tick and nothing else. Progress indicators are used sparingly — only when a check takes more than ~300ms.

section 4 of 4

Design rationale

Why this architecture, not another.

choice
Three tiers, not one toast

A single toast system collapses urgency — a "file saved" toast and a "blocking error" toast look identical. Three surfaces let urgency be communicated through position and persistence, not just colour. The status bar is always visible; the banner demands a dismiss; the coach is an invitation. (Principle 5)

choice
Banners don't stack

A new banner replaces the old one. Stacked notifications train users to dismiss everything without reading. One banner at a time means one thing to pay attention to. The old event is still in the timeline if needed.

choice
Error messages are tasks, not codes

"Can't write: 04-how-it-works.md — file is locked" is a task. "EACCES: permission denied, open '/…/04-how-it-works.md'" is not. Every error message must answer: what failed, why, and what to do. (DR-003)

choice
No telemetry, no error reporting

DR-006 rules out any server-side error collection. Errors are surfaced to the user only. The plugin can offer a "copy error to clipboard for GitHub issue" action, but never auto-submits anything. User owns all data, including errors.
