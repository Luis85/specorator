design rationale for this flow

Why we made these choices

choice
Yellow, not red

Red says "you broke something." Yellow says "let's pause." The user hasn't done anything wrong — they've reached a checkpoint. Tone matters.

choice
Auto-check, not manual

The plugin counts rules and parses links. The user never has to "press check." Gates are continuously-evaluated; the button just lets you advance once they pass.

choice
Each failure has a fix-button

A blocked user is a frustrated user. Every failure must answer "and what do I do?" — open a file, add a section, mark a question. Never just a "your gate failed" wall.

choice
Gate ceremony > hand-off ceremony

A hand-off is a step. A gate is a lane change. The cross-fade is slower (400ms vs 250ms) and the cockpit recolors. Reserved emphasis where it matters.

decision · OQ-3 resolved
2026-05-01 · accepted
Auto-pass for solo; explicit sign-off for team

Choice: Gate passes automatically when all checklist items are ticked, with the "Cross the gate" button as the user's deliberate intent-signal. Solo builders get no extra friction. In team mode, the Define→Build gate additionally requires a peer-review tick (see edge case B in this flow) — the PR merge is the sign-off.

Why auto-pass for solo: A solo builder is already accountable to themselves. Adding a mandatory sign-off step on a gate they alone control adds ceremony without safety. The explicit button click is enough intent-signal. If users fly through gates without reading, the answer is better gate copy — not a second confirmation.

Why explicit for team: A gate crossing that affects a collaborator's work needs a human witness. The PR template carries the checklist; merging is the sign-off. No extra UI required — it piggybacks on an existing ritual.
