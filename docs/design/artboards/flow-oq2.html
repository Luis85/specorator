Cockpit empty state — what do you show when nothing is active?

Three candidates were on the table: show a feed of everything needing attention across all features, return to the last-focused feature, or show a true empty state with a "pick a feature" prompt. This flow documents the decision and shows the three states the cockpit must handle.

decision · OQ-2 resolved
2026-05-01 · accepted
Last-focused feature, with a feed fallback when truly empty

Choice: When the user opens Obsidian and no feature is explicitly selected, the cockpit restores the last-focused feature. If no feature has ever been focused (new vault, or all features archived), it shows a minimal empty state with one action: "Start a feature."

Why not the feed: A feed of all features needing attention is useful — but it's a portfolio view, not a cockpit. The cockpit is designed for single-feature depth. The feed is better as a dedicated view (Cockpit D) than as the default empty state. Mixing them creates a context-switching cost every time you open Obsidian.

Why last-focused over empty: Most users work on one feature at a time. Restoring their last context is almost always correct. The empty state is a genuine edge case — new vault or post-archive — not the common path.

why not the feed
Feed = portfolio view, not empty state

Cockpit D (the feed) is a deliberate choice — a view the user switches to. Making it the default empty state means every vault-open starts with a context switch. The cockpit is for depth; the feed is for breadth. They serve different moments.

why last-focused
Context restoration = zero friction re-entry

The biggest competitor to Specorator is "I'll just keep it in my head." Context restoration has to be instant — if opening the cockpit costs a click to pick a feature, some users will stop opening it. Last-focused is almost always right; the edge cases (picker, earned empty) are rare.
