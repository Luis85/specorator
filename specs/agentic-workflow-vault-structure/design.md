## Summary

This document resolves the two open design questions from the requirements stage and defines the canonical `workflow-state.md` frontmatter schema, stage artifact stub templates, and migration detection behavior.

---

## Decision 1 — ADR storage location

**Question:** Should ADRs be stored per-feature (`specs/{slug}/adr/`) or globally (`docs/adr/`)?

**Decision:** Global — `docs/adr/`.

**Rationale:**

- The repository already has four ADRs under `docs/adr/`, establishing a global convention.
- ADRs document architectural decisions that affect the whole system, not a single feature. Scattering them per-feature makes them harder to discover and cross-reference.
- The agentic-workflow template does not require per-feature ADR folders; that would be a Specorator extension not compatible with the upstream convention.
- Per-feature ADRs may be revisited in v2.0 if the need emerges from actual usage.

**Impact:** No change to the plugin or vault structure. No new folder type required.

---

## Decision 2 — Existing `features/` vault handling

**Question:** How should the plugin detect and handle vaults that already have a `features/` structure?

**Decision:** Detect on load, show a persistent notice, take no automatic action.

**Rationale:**

- Silent migration is unsafe — user-created file names or content may not map cleanly to the new slug list.
- Offering an explicit command ("Migrate features/ to specs/") is deferred to a post-v1 issue to keep v1 scope tight.
- A notice is enough to inform the user without surprising them with moved files.

**Behavior (v1):**

On plugin load, if `features/` exists as a vault folder and `specs/` does not, the plugin calls `IBridge.showNotice` with the message:

> "Specorator: this vault uses the old `features/` folder. Please rename it to `specs/` or update the Specs folder setting."

No automatic rename or migration is performed. The notice is shown once per load.

**Impact:** Add a one-time detection call in `SpecoratorPlugin.onload`. No domain or repository changes needed.

## Canonical `workflow-state.md` frontmatter schema

This is the complete YAML frontmatter the plugin writes when creating a new feature. Fields are listed in agentic-workflow document order; plugin-internal fields follow.

```yaml
---
id: {uuid}
feature: "{title}"
area: "{AREA}"
current_stage: idea
status: draft
last_updated: {ISO-8601 date}
last_agent: ""
artifacts:
  idea: complete
  research: pending
  requirements: pending
  design: pending
  spec: pending
  tasks: pending
  implementation-log: pending
  test-plan: pending
  test-report: pending
  review: pending
  release-notes: pending
  retrospective: pending
createdAt: {ISO-8601 datetime}
updatedAt: {ISO-8601 datetime}
slug: {slug}
---
```

### Field notes

| Field | Notes |
|---|---|
| `id` | UUIDv4 generated by the plugin; stable across renames. |
| `feature` | Display title. This is the canonical title field; agentic-workflow reads `feature`, not `title`. The `title:` field written by previous builds must not be written going forward. |
| `area` | 3–5 uppercase letters (REQ-AVS-006). Derived from the first letters of the slug if not user-provided (e.g. `dark-mode` → `DM`). |
| `current_stage` | One of the 12 agentic-workflow stage slugs from `FEATURE_STEPS`. Derived from `currentStep`; not an independent field — always kept in sync. |
| `status` | `draft \| active \| archived \| abandoned`. Matches `FeatureStatus`. |
| `last_updated` | ISO-8601 date string (`YYYY-MM-DD`), not full datetime. Updated on every write. |
| `last_agent` | Empty string in v1. Reserved for agentonomous integration in v2.0. |
| `artifacts` | YAML block map. One entry per stage. Values: `pending \| in-progress \| complete \| skipped \| blocked`. On creation: `idea: complete`, all others `pending`. Updated when a stage file is created or the user advances. |
| `createdAt` / `updatedAt` | Full ISO-8601 datetime; plugin-internal for domain reconstitution. |
| `slug` | Plugin-internal; used to resolve the folder path without directory scanning. |

### `artifacts` map — initial value on feature creation

```yaml
artifacts:
  idea: complete
  research: pending
  requirements: pending
  design: pending
  spec: pending
  tasks: pending
  implementation-log: pending
  test-plan: pending
  test-report: pending
  review: pending
  release-notes: pending
  retrospective: pending
```

When the user advances to stage N, the `artifacts` entry for that stage changes from `pending` to `in-progress`. When the stage file is created, it moves to `in-progress`. Marking complete is a manual or future-agent action.

### Stage advancement rule

When the user advances from step N to step N+1:

1. Update `current_stage` to the new stage slug.
2. Update `artifacts[new_stage]` from `pending` to `in-progress`.
3. Create `specs/{slug}/{new_stage}.md` with the stub template (see below) if the file does not already exist. If it exists, show a notice and skip creation (REQ-AVS-005).
4. Write the updated `workflow-state.md`.

## Stage artifact stub templates

When the plugin creates a stage file it writes a minimal stub containing only the stage header. This keeps the file useful without the plugin and compatible with agentic-workflow agent prompts.

General stub pattern:

```markdown
---
stage: {stage-slug}
feature: {slug}
status: in-progress
created: {YYYY-MM-DD}
---

<!-- {Stage display name} artifact for {feature title}. -->
```

Stage display names for headings:

| Slug | Display name |
|---|---|
| `idea` | Idea |
| `research` | Research |
| `requirements` | Requirements |
| `design` | Design |
| `spec` | Specification |
| `tasks` | Tasks |
| `implementation-log` | Implementation log |
| `test-plan` | Test plan |
| `test-report` | Test report |
| `review` | Review |
| `release-notes` | Release notes |
| `retrospective` | Retrospective |

The `idea` stage file is created at the same time as `workflow-state.md` (feature creation), not on advancement, since the idea is always the starting point.

## Serializer changes required

The current `serializeFeature` function in `FeatureRepository.ts` needs the following updates:

1. **Remove `title:`** — only `feature:` should be written going forward.
2. **Replace `artifacts: []`** with the full YAML block map (12 entries, initial status derived from `currentStep`).
3. **Add `area:`** — derive from slug initials if the domain model does not carry an explicit area (see REQ-AVS-006). Store `area` on the `Feature` domain object or as a repository-level concern.
4. **Change `last_updated`** from full datetime to date-only (`YYYY-MM-DD`).
5. **Deserializer** — update `parseFrontmatter` to read the `artifacts` block map (currently stored as a flat `artifacts: []` value).

These changes are implementation-stage work. The design here defines the target schema; the spec stage will define the exact function signatures and acceptance tests.

## Out of scope for this design

- Per-feature ADR folders (deferred to v2.0).
- Automated `features/` → `specs/` migration command (deferred to post-v1).
- `traceability.md` generation (explicitly out of scope per requirements).
- Obsidian mobile vault path handling (desktop only in v1).
